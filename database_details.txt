# Database Schema & Data Relationships - Email Dashboard

This document provides a detailed overview of the MongoDB collections and the relational structure of the Email Dashboard API.

---

## 1. Collections Overview

### users
Stores user accounts, roles, and access permissions.
- **Fields**: `email`, `full_name`, `password_hash`, `password` (plaintext, testing only), `role` (`admin` / `manager` / `employee`), `phone_number` (optional), `permissions` (`{"dashboard": []}` by default), `created_at`.

### clients
Stores client information.
- **Fields**: `client_id` (Primary Key, e.g. `CL-001`), `name`, `country`, `email`, `whatsapp_no`, `client_ref_no`, `client_link`, `bank_account`, `affiliation`, `total_orders` (denormalised count), `client_handler` (links to `users.full_name`), `created_at`.

### manuscripts
Stores records for manuscripts submitted by clients.
- **Fields**: `manuscript_id` (Primary Key, e.g. `MS-CL-001-1`), `title`, `order_type` (`writing` / `modification` / `proofreading`), `client_id` (Foreign Key → `clients`), `client_ref_no`, `created_at`.

### orders
Stores order details linked to manuscripts and clients.
- **Fields**: `order_id` (Primary Key, e.g. `ORD-2024-001`), `s_no`, `order_date`, `client_id` (Foreign Key → `clients`), `manuscript_id` (Foreign Key → `manuscripts`), `client_ref_no`, `order_type`, `index` (`SCI` / `Scopus` / `ESCI`), `rank` (`Q1`–`Q4`), `currency` (`USD` / `INR`), `total_amount`, `writing_amount`, `modification_amount`, `po_amount`, `writing_start_date`, `writing_end_date`, `modification_start_date`, `modification_end_date`, `po_start_date`, `po_end_date`, `payment_status` (`pending` / `partial` / `paid`), `assigned_to` (links to `users._id`), `remarks`, `created_at`, `updated_at`.

### payments
Stores payment transactions linked to orders and clients, supporting up to 3 payment phases.
- **Fields**: `client_ref_number`, `client_id` (Foreign Key → `clients`), `order_id` (Foreign Key → `orders`), `phase` (1, 2, or 3), `amount`, `payment_received_account`, `payment_date`, `phase_1_payment`, `phase_1_payment_date`, `phase_2_payment`, `phase_2_payment_date`, `phase_3_payment`, `phase_3_payment_date`, `status` (`pending` / `paid`), `created_at`.

### tokens
Stores active session JWTs.
- **Fields**: `user_email` (Foreign Key → `users`), `token`, `created_at`.

### otps
Temporary storage for 2FA login codes (Admin & Manager roles only).
- **Fields**: `email` (Foreign Key → `users`), `otp`, `created_at`.

## 2. Data Relationships

The system maintains referential integrity through logical IDs (e.g., `client_id`) rather than MongoDB ObjectIDs for easier cross-service tracking.

```
erDiagram
    Users ||--o{ Clients : "handled by"
    Users ||--o{ Orders : "assigned to"
    Users ||--o{ Tokens : "owns"
    Users ||--o{ OTPs : "receives"

    Clients ||--o{ Manuscripts : "submits"
    Clients ||--o{ Orders : "places"
    Clients ||--o{ Payments : "makes"

    Manuscripts ||--o{ Orders : "referenced in"

    Orders ||--o{ Payments : "has"
```

### Relationship Details:

1. **User → Client**:
   - Relationship: One-to-Many.
   - Link: `users.full_name` matches `clients.client_handler`.
   - Purpose: Employees can only view/manage clients assigned to them; Managers/Admins see all.

2. **User → Order**:
   - Relationship: One-to-Many.
   - Link: `users._id` (as string) matches `orders.assigned_to`.
   - Purpose: Tracks which employee is handling a specific order.

3. **Client → Manuscript**:
   - Relationship: One-to-Many.
   - Link: `clients.client_id` matches `manuscripts.client_id`.

4. **Client → Order**:
   - Relationship: One-to-Many.
   - Link: `clients.client_id` matches `orders.client_id`.

5. **Manuscript → Order**:
   - Relationship: One-to-One / One-to-Many (depending on revisions).
   - Link: `manuscripts.manuscript_id` matches `orders.manuscript_id`.

6. **Order → Payment**:
   - Relationship: One-to-Many (up to 3 payment phases).
   - Link: `orders.order_id` matches `payments.order_id`.
   - Purpose: Tracks multiple payment phases for a single order.

7. **Client → Payment**:
   - Relationship: One-to-Many.
   - Link: `clients.client_id` matches `payments.client_id`.
   - Purpose: Supports country-level financial aggregation for pie charts.

## 3. ID Conventions

- **Client ID**: `CL-XXX` (e.g., `CL-001`)
- **Manuscript ID**: `MS-[Client_ID]-[Seq_No]` (e.g., `MS-CL-001-1`)
- **Order ID**: `ORD-[Year]-[Seq_No]` (e.g., `ORD-2024-001`)
- **Client Ref No**: `REF-XXXX` (e.g., `REF-2001`) — used as a human-readable cross-reference across clients, manuscripts, and payments.

## 4. Key API Schemas

### DashboardOrderResponse
A flattened, joined view of client + order + payment data returned by the main dashboard endpoint.
- Merges fields from: `clients` (country, email, whatsapp_no, client_link, bank_account, affiliation), `orders` (amounts, dates, status, index, rank), `payments` (phase_1/2/3 payment amounts and dates).

### DashboardUpdate
Used to update client, order, and payment data in a single request.
- **Client fields**: `client_location` (maps to `country`), `client_Email`, `client_whatsapp_number`, `client_link`, `bank_account`, `client_affiliations`.
- **Order fields**: `order_date`, `ref_no`, `order_type`, `index`, `rank`, `currency`, `total_amount`, `writing_amount`, `modification_amount`, `po_amount`, all phase start/end dates, `payment_status`, `remarks`.
- **Payment fields**: `phase_1_payment`, `phase_1_payment_date`, `phase_2_payment`, `phase_2_payment_date`, `phase_3_payment`, `phase_3_payment_date`.

### ClientDetailResponse
Per-client financial summary returned inside `/users/me/details`.
- Aggregated fields: `total_amount`, `writing_amount`, `modification_amount`, `po_amount`, `paid_amount`, `remaining_amount`, `payment_status`.

### UserDetailResponse
Full user detail object returned by `/users/me/details`.
- Includes: all `UserBase` fields + `handled_clients: list[ClientDetailResponse]`, `dashboard_stats: DashboardStats`, `country_split: dict[str, float]` (used for pie chart visualisation).

### DashboardStats
Top-level KPIs on the dashboard.
- Fields: `overall_amount`, `overall_amount_percentage`, `total_clients`, `total_clients_percentage`, `pending_count`, `pending_count_percentage`, `reject_count`, `reject_count_percentage`.

## 5. Role-Based Access Control (RBAC)

| Role     | Clients Visible        | Orders Visible                | Payments Visible         |
|----------|------------------------|-------------------------------|--------------------------|
| Admin    | All clients            | All orders                    | All payments             |
| Manager  | All clients            | All orders                    | All payments             |
| Employee | Only assigned clients  | Orders for assigned clients   | Payments for assigned clients |

- Admins and Managers require **OTP verification** (2FA via email) on login before receiving a JWT.
- Employees receive a JWT directly on login.
